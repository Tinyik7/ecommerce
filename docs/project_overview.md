## 1. Проблема

- **Разрозненный онлайн-шопинг** — локальные бренды одежды и аксессуаров публикуют ассортимент в мессенджерах и соцсетях, поэтому пользователю приходится прыгать между каналами и вручную уточнять наличие размеров.
- **Низкая прозрачность статуса заказа** — нет единой точки, где видно актуальное наличие, стоимость доставки и рекомендации по размеру.

## 2. Целевая аудитория

| Сегмент | Потребности | Как решаем |
| --- | --- | --- |
| Студенты и молодые специалисты 18–28 | Быстрые покупки со смартфона, рассрочка МПС, push-уведомления о дропах | Нативное Flutter-приложение, офлайновый кеш каталога, пуши через FCM (в дорожной карте) |
| Владельцы шоурумов | Единый каталог, мгновенное обновление остатков, возможность продвигать коллекции | Бэкенд на Spring + Swagger, админ API для загрузки карточек (JSON или multipart) |
| Постоянные клиенты | Персональные рекомендации, избранное, история заказов | Локальная БД + фильтры, избранное, сохранённые корзины |

## 3. Новизна и ценность

1. **Единый стек**: Flutter + Spring Boot + PostgreSQL + локальный SQLite кеш → каталоги работают даже офлайн (приложение показывает сохранённые товары и корзину).
2. **Двусторонняя синхронизация**: избранное и корзина синхронизируются с сервером и локально, есть тестовая Postman-коллекция и Swagger UI (`/swagger`).
3. **Безопасность**: JWT-авторизация + разграничение ролей (покупатель/администратор) для доступа к панели управления.
4. **Гибкая фильтрация**: поиск по названию/описанию, категории, цене, наличию, сортировки по новизне/цене/рейтингу как на фронте, так и на API (`GET /api/v1/products?minPrice=50&sortBy=price`).
5. **Документированность**: все критерии из ведомости вынесены в отдельные файлы (`docs/*`), чтобы быстро закрыть требования на предзащите.
